# Phase: Execution

**Purpose:** Implement the plan, create deliverables, and build the solution.

**Duration:** Typically 40-60% of total project time

**Question to Answer:** *Can we build this according to plan?*

---

## Your Role in This Phase

You are an **Implementer/Developer/Creator**. Your job is to:
- Execute the plan created in the planning phase
- Create high-quality deliverables
- Follow project standards and conventions
- Document as you go
- Maintain quality throughout

## Phase Entry Checklist

Before starting this phase, confirm:
- [ ] Planning phase is complete
- [ ] Plan is approved by user
- [ ] You have all necessary context
- [ ] Any specialist templates are loaded

## Execution Principles

### 1. Follow the Plan

```
EXECUTION SEQUENCE:

1. Review the implementation plan
2. Execute steps in order
3. Document progress
4. Flag deviations immediately
5. Get approval for changes

NEVER:
- Skip steps without approval
- Add scope without discussion
- Ignore the agreed approach
```

### 2. Quality Gates

```
AFTER EACH MAJOR STEP:

□ Code/content quality check
□ Self-review
□ Documentation updated
□ Tests written (if applicable)
□ Ready for next step
```

### 3. Incremental Delivery

```
PREFER SMALL, COMPLETE INCREMENTS:

Instead of: "Working on everything at once"
Do: "Complete Feature A → Review → Complete Feature B"

This allows for:
- Early feedback
- Course correction
- Reduced risk
- Better quality
```

## Execution by Workflow Type

### Software Development

```
ITERATIVE CYCLE:

1. SETUP (if not done)
   □ Initialize project structure
   □ Install dependencies
   □ Configure tools (linting, formatting)
   □ Set up version control

2. IMPLEMENT FEATURE
   □ Write code following standards
   □ Add tests (unit/integration)
   □ Add inline documentation
   □ Self-review
   □ Run tests locally

3. COMMIT
   □ Clear commit message
   □ Atomic changes
   □ Reference issues if applicable

4. REPEAT
   □ Next feature
   □ Continue until complete

STANDARDS TO FOLLOW:
- Code style from project conventions
- Error handling best practices
- Security considerations
- Performance awareness
```

### Content Creation

```
DRAFTING PROCESS:

1. FIRST DRAFT (Fast, messy)
   □ Get ideas down
   □ Don't edit while writing
   □ Complete all sections
   □ Mark areas needing research

2. SECOND DRAFT (Structure)
   □ Organize flow
   □ Add transitions
   □ Ensure logical progression
   □ Check completeness

3. THIRD DRAFT (Refine)
   □ Improve word choice
   □ Cut unnecessary content
   □ Enhance clarity
   □ Add examples/data

4. POLISH
   □ Grammar and spelling
   □ Formatting consistency
   □ SEO optimization (if applicable)
   □ Final review
```

### Research & Analysis

```
RESEARCH PROCESS:

1. SYSTEMATIC SEARCH
   □ Follow search strategy from plan
   □ Document all sources
   □ Save relevant materials
   □ Note credibility of sources

2. DATA COLLECTION
   □ Extract key information
   □ Organize by theme
   □ Note contradictions
   □ Identify gaps

3. ANALYSIS
   □ Apply analytical framework
   □ Look for patterns
   □ Synthesize findings
   □ Form conclusions

4. DOCUMENTATION
   □ Write findings
   □ Include citations
   □ Add visualizations (if helpful)
   □ Note limitations
```

### Business Strategy

```
DEVELOPMENT PROCESS:

1. DATA GATHERING
   □ Collect market data
   □ Analyze financials
   □ Interview stakeholders
   □ Research competitors

2. ANALYSIS
   □ Apply strategic frameworks
   □ Model scenarios
   □ Evaluate options
   □ Identify trade-offs

3. SYNTHESIS
   □ Develop recommendations
   □ Create action plans
   □ Define metrics
   □ Build business case

4. VALIDATION
   □ Sanity check assumptions
   □ Test with stakeholders
   □ Refine based on feedback
```

## Handling Deviations

### When the Plan Needs to Change

```
1. IDENTIFY the deviation
   "The original plan assumed X, but we're finding Y"

2. ASSESS impact
   - Time impact?
   - Quality impact?
   - Scope impact?

3. PROPOSE options
   Option A: [Adjust approach]
   Option B: [Extend timeline]
   Option C: [Reduce scope]

4. GET USER APPROVAL
   "Given this issue, I recommend [option]. Agreed?"

5. DOCUMENT the change
   Add to execution log and update plan
```

### Common Issues & Solutions

| Issue | Solution |
|-------|----------|
| Taking longer than planned | Communicate early, propose scope/timeline adjustment |
| Technical blocker | Research alternatives, ask for help if needed |
| Unclear requirements | Pause, document questions, get clarification |
| Scope creep | Flag immediately, discuss impact, get approval |
| Quality concerns | Don't rush, propose additional review time |

## Common Pitfalls to Avoid

❌ **Don't:**
- Rush to finish without checking quality
- Leave "TODO" comments without documenting
- Skip tests to save time
- Commit broken code
- Ignore edge cases

✅ **Do:**
- Test as you go
- Document assumptions
- Communicate progress regularly
- Ask for help when stuck
- Keep the user informed

## Transition to Review

When this phase is complete:

1. **Summarize work:**
   "I've completed [summary of work]. Here's what was delivered..."

2. **Present deliverables:**
   "You can find the work in [location]. Let me walk you through it..."

3. **Explain next steps:**
   "Now we'll move to review phase to ensure quality. Ready?"

4. **Load next phase:**
   Read `phases/review.md` and follow those instructions.
